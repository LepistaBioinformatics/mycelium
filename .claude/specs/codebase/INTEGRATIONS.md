## PostgreSQL (Diesel ORM)

### Connection & Pooling
- **Library**: Diesel 2.x with actix integration
- **Pool Provider**: `DieselDbPoolProvider` in `adapters/diesel/src/repositories/config.rs`
- **Connection String**: Via environment variable or config file
- **Pool Type**: `PgConnection` from Diesel

### Schema & Migrations
- **DDL File**: `adapters/diesel/sql/up.sql` (complete schema in one file)
- **Diesel Config**: `adapters/diesel/diesel.toml`
- **Schema Definition**: `adapters/diesel/src/schema.rs` (auto-generated by Diesel CLI)

### Database Tables (from `sql/up.sql`)
- `tenant` — Multi-tenancy support (id UUID, name VARCHAR, meta JSONB, status JSONB[])
- `account` — Accounts with lifecycle flags (is_active, is_deleted, is_archived, is_checked)
- `user` — Users with email, MFA state (mfa JSONB)
- `identity_provider` — OAuth/SAML provider mappings
- `token` — Session/auth tokens (email_confirmation, password_reset types)
- `message` — Email message queue (subject, body HTML, status)
- `webhook` — Event webhooks (trigger JSONB, url)
- `guest_user`, `guest_role` — Guest/anonymous access
- `profile` — User profiles with ownership hierarchy
- `licensed_resource` — Resource licensing
- `account_tag`, `tenant_tag` — Tagging/categorization
- `error_code` — Application error definitions

### Diesel Patterns
```rust
use diesel::prelude::*;
let existing_user = user_model::table
    .filter(user_model::email.eq(email))
    .select(UserModel::as_select())
    .first::<UserModel>(conn)
    .optional()?;

// Insert with returning
diesel::insert_into(user_model::table)
    .values(new_user)
    .returning(UserModel::as_returning())
    .get_result::<UserModel>(conn)?;

// Transaction
conn.transaction(|conn| {
    // Multiple operations
    Ok(result)
})?;
```

### Repository Pattern
- Separate files per CRUD operation per entity
- Pattern: `UserRegistration`, `UserFetching`, `UserUpdating`, `UserDeletion`
- Trait implementations via Shaku `#[derive(Component)]`
- Example: `/mnt/external/thirdparty-projects/mycelium/adapters/diesel/src/repositories/user/user_registration.rs`

## Redis (KV Store)

### Library & Configuration
- **Library**: `redis` 0.27 with `tokio-comp` feature
- **Adapter**: `adapters/kv_db/` standalone crate
- **Client Provider**: `SharedClientProvider` trait from `adapters/shared/`

### Ports & Repositories
- **KVArtifactRead**: Fetch encoded artifacts from Redis
  ```rust
  pub struct KVArtifactReadRepository {
      #[shaku(inject)]
      client: Arc<dyn SharedClientProvider>,
  }
  
  async fn get_encoded_artifact(&self, key: String) 
      -> Result<FetchResponseKind<String, String>, MappedErrors>
  ```
- **KVArtifactWrite**: Store encoded artifacts to Redis
  - Likely uses `set`, `expire` commands
  - Location: `adapters/kv_db/src/repositories/kv_artifact_write.rs`

### Usage Pattern
- Compressed/encoded data stored as strings
- Key-based access via `get()` / `set()` commands
- Used for: Session cache, artifact storage, message queuing
- Error handling: `fetching_err` for missing keys

Example from code (`kv_artifact_read.rs` lines 38-42):
```rust
let rv: Value = connection.get(key.to_owned()).map_err(|err| {
    fetching_err("Error on get redis artifact")
})?;

if Value::Nil == rv {
    return Ok(FetchResponseKind::NotFound(Some(key)));
}
```

## OAuth2 & OIDC (External Identity Providers)

### Supported Providers
- **Google** (OAuth2 + OIDC)
- **Microsoft** (Azure AD + OIDC)
- **Auth0** (OIDC)
- **Custom OIDC providers**

### Configuration
- **Type**: `ExternalProviderConfig` in `lib/http_tools/src/models/external_providers_config.rs`
- **Fields**:
  - `issuer` — Token issuer (async SecretResolver)
  - `jwks_uri` — JWKS endpoint for key validation
  - `audience` — Client ID/audience for token validation
  - `discovery_url` — OpenID Connect discovery endpoint
  - `user_info_url` — User info endpoint
  
- **Storage**: Via `SecretResolver<String>` for sensitive fields
- **Async**: Configured via `.async_get_or_error().await?`

### OAuth2 Implementation
- **Library**: `oauth2` 4.4 (client-side OAuth2 flow)
- **JWT Validation**: `jsonwebtoken` 10 with `aws_lc_rs` cryptography
- **Token Format**: JWT with RS256 (RSA signatures)
- **Middleware**: `actix-web-httpauth` 0.8.0 for HTTP auth parsing

### Example Configuration
```rust
pub struct ExternalProviderConfig {
    pub issuer: SecretResolver<String>,           // e.g., https://accounts.google.com
    pub jwks_uri: SecretResolver<String>,         // JWKS cert endpoint
    pub audience: SecretResolver<String>,         // Client ID
    pub discovery_url: Option<SecretResolver<String>>,  // .well-known/openid-configuration
    pub user_info_url: Option<SecretResolver<String>>,  // Userinfo endpoint
}
```

## Email (Lettre SMTP)

### Library & Configuration
- **Library**: `lettre` 0.11 (async SMTP client)
- **Adapter**: `adapters/notifier/` standalone crate
- **TLS**: Via `openssl` >= 0.10.70

### Repositories
- **RemoteMessageWrite**: Send email via SMTP
  - Implementation: `adapters/notifier/src/repositories/remote_message_sending.rs`
  - Uses: `lettre::Message::builder()` + `Transport::send()`

- **LocalMessageWrite**: Store messages in database before sending
  - Allows async dispatch and retry logic

### SMTP Configuration
- **Type**: `SmtpConfig` in `adapters/notifier/src/models/config.rs`
- **Fields**: Username, password (from Vault), host, port, TLS settings
- **Credentials**: Resolved via `SecretResolver` from Vault
  ```rust
  let username = smtp_config.username.async_get_or_error().await?;
  let password = smtp_config.password.async_get_or_error().await?;
  let credentials = Credentials::new(username, password);
  ```

### Email Sending Pattern
From `RemoteMessageSendingRepository`:
```rust
let email = LettreMessage::builder()
    .from(message.from.email().parse().unwrap())
    .to(message.to.email().parse().unwrap())
    .subject(message.subject)
    .header(ContentType::TEXT_HTML)
    .body(message.body)
    .unwrap();

connection.send(&email)?;
```

### Message Template
- **Type**: `Message` DTO with `FromEmail`, `to`, `subject`, `body` (HTML)
- **Ports**:
  - `LocalMessageWrite` — Store to database
  - `RemoteMessageWrite` — Send via SMTP

## OpenTelemetry (Observability)

### Libraries
- **opentelemetry** 0.31 (traces, metrics, logs)
- **opentelemetry_sdk** 0.31 with `rt-tokio`
- **opentelemetry-otlp** 0.31 (OTLP exporter)
  - Supports gRPC and HTTP protocols
  - Endpoint: `http://host:port` for HTTP, same for gRPC
- **tonic** 0 (gRPC support)
- **tracing-opentelemetry** 0.32 (bridge to tracing)
- **tracing-appender** 0.2 (file logging)

### Configuration
- **Type**: `LoggingConfig` in `ports/api/src/models/`
- **Targets**:
  - `LoggingTarget::File { path }` — Write to file
  - `LoggingTarget::Collector { name, protocol, host, port }` — OTLP endpoint
  - Default: STDERR

### Tracing Setup
From `ports/api/src/otel.rs` (lines 12-160):
```rust
let tracer_provider = opentelemetry_sdk::trace::SdkTracerProvider::builder()
    .with_resource(resource.clone());

match protocol {
    Protocol::Grpc => {
        let trace_exporter = opentelemetry_otlp::SpanExporter::builder()
            .with_tonic()
            .with_endpoint(traces_address)
            .with_timeout(Duration::from_secs(10))
            .build()?;
        tracer_provider.with_batch_exporter(trace_exporter)
    }
    _ => {
        let trace_exporter = opentelemetry_otlp::SpanExporter::builder()
            .with_http()
            .with_endpoint(traces_address)
            .build()?;
        tracer_provider.with_batch_exporter(trace_exporter)
    }
}
```

### Instrumentation
- **Macro**: `#[tracing::instrument]` on all async functions
- **Span Fields**: `myc.router.req_id`, `myc.router.req_method`, `myc.router.res_status`, `myc.router.res_duration`
- **Log Levels**: `RUST_LOG` environment variable
- **Format**: JSON (optional) via `tracing-subscriber::fmt`

## MCP Server Integration (Claude AI)

### Library & Protocol
- **Protocol**: JSON-RPC 2.0 over HTTP
- **Port**: Served on `/mcp` endpoint
- **Framework**: Actix-web via `ports/api/src/mcp/`

### Endpoints & Methods
- **URL**: `/mcp` (HTTP POST)
- **Methods**:
  - `initialize` — Initialize MCP session
  - `tools/list` — List available tools from OpenAPI schema
  - `tools/call` — Execute a tool (call downstream service)

### Implementation
From `ports/api/src/mcp/endpoints.rs`:
```rust
#[post("")]
async fn mcp_endpoint(
    http_req: HttpRequest,
    body: web::Bytes,
    tools_registry: web::Data<ServiceOpenApiSchema>,  // Tool definitions
    client: web::Data<Client>,                         // HTTP client
    api_config: web::Data<ApiConfig>,
) -> impl Responder {
    let request: JsonRpcRequest = serde_json::from_slice(&body)?;
    
    match request.method.as_str() {
        "initialize" => handle_initialize(&request),
        "tools/list" => handle_list_tools(&request, &tools_registry),
        "tools/call" => handle_call_tool(&http_req, &request, ...).await,
        _ => error_response(...),
    }
}
```

### Tool Registry
- **Type**: `ServiceOpenApiSchema`
- **Source**: Built from OpenAPI documentation of routes
- **Location**: `ports/api/src/openapi_processor.rs`
- **Initialization**: `initialize_tools_registry()` from main.rs (line 16)

### MCP Handlers
- **Location**: `ports/api/src/mcp/handlers/`
- **Responsibilities**:
  - Extract tool parameters
  - Call downstream service
  - Format response as JSON-RPC
  - Handle errors and edge cases

## Service Discovery & Routing

### Service Definition
- **Type**: Service model in `adapters/service/`
- **Fields**: Name, URL, allowed methods, security groups, tags
- **Storage**: MemDb (in-memory repository)
- **Loaded**: At startup via configuration

### Route Matching
- **From**: Service + request path
- **To**: Downstream URL
- **Pattern Matching**: Via wildmatch for path patterns
- **Context Injection**: Security group, user scope, permissions
